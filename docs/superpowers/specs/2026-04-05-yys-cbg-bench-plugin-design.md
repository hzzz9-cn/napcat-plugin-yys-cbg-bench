# 阴阳师藏宝阁分析 NapCat 插件设计文档

日期：2026-04-05

## 1. 背景

现有仓库 `yys-cbg-bench` 是一个长期未更新的 Python CLI/EXE 工具，核心能力是：

1. 读取阴阳师藏宝阁商品链接。
2. 拉取藏宝阁商品详情和辅助配置。
3. 分析账号资源、图鉴、御魂与部分配装结论。
4. 输出文本和图片报告。

本次不再延续 CLI 或 EXE 方向，而是基于 `napcat-plugin-yys-cbg-bench` 模板，重构为一个 NapCat 插件。插件的主要使用场景是群聊里出现藏宝阁链接时自动触发分析，并由机器人回发一张数据海报风格的图片。

## 2. 目标

### 2.1 本次目标

1. 基于 NapCat 插件模板实现原生 TypeScript 插件，不依赖外部 Python 子进程。
2. 群聊中出现合法藏宝阁商品链接时自动触发分析。
3. 机器人先发送“正在分析”提示，再回发结果图片。
4. 报告图片采用“数据海报”视觉方向。
5. 图片通过 HTML 模板渲染后由 Playwright 截图生成。
6. 生成的图片通过插件静态资源路径对外提供访问。
7. 报告文件保留 24 小时，自动清理。
8. WebUI 作为管理后台存在，用于配置、群管理、查看状态与最近报告。
9. 错误响应不暴露机器目录、绝对路径、调用栈等敏感信息。

### 2.2 非目标

1. 不保留原有 CLI 入口。
2. 不保留 EXE 打包能力。
3. 不以 WebUI 手动输入链接作为主要业务入口。
4. 不在本次提供完整分析 JSON 对外接口。
5. 不在本次引入独立外部 HTTP 服务进程。

## 3. 用户体验

### 3.1 群聊触发流程

1. 用户在群聊发送消息。
2. 插件扫描消息正文中的阴阳师藏宝阁商品链接。
3. 若匹配到合法链接且当前群启用插件，则进入分析流程。
4. 机器人立即回复“正在分析这条藏宝阁链接，请稍等”。
5. 分析成功后，机器人发送图片消息。
6. 分析失败时，机器人发送简短错误提示，不暴露内部路径和堆栈信息。

### 3.2 WebUI 角色

WebUI 不是主业务入口，而是插件的管理后台，负责：

1. 查看插件运行状态。
2. 管理全局配置。
3. 管理群启用状态。
4. 查看最近生成的报告与最近错误摘要。
5. 手动触发过期报告清理。

## 4. 实现方案对比

### 4.1 方案 A：模板中调用旧 Python 脚本

优点：

1. 上手快。
2. 能最快复用旧代码。

缺点：

1. 插件运行依赖 Python 环境，部署复杂。
2. TS 插件和 Python 脚本形成双运行时耦合。
3. 后续适配新数据、WebUI 管理和错误处理都会变得混乱。

### 4.2 方案 B：基于模板做原生插件重构

优点：

1. 与模板的生命周期、配置体系、消息处理和 WebUI 完全一致。
2. 部署路径简单，只需 NapCat 插件环境和 Node 依赖。
3. 更适合后续继续补数据规则、报告样式和管理后台能力。

缺点：

1. 需要一次性将旧分析逻辑迁到 TypeScript。

### 4.3 方案 C：插件壳 + 内置本地 HTTP 服务

优点：

1. 后续可独立演化成 API 服务。

缺点：

1. 对当前模板来说过重。
2. 引入额外进程与运维复杂度。
3. 与“群聊自动触发”为主入口的诉求不匹配。

### 4.4 选择

采用方案 B：基于 `napcat-plugin-yys-cbg-bench` 模板做原生 NapCat 插件重构。

## 5. 模板内架构设计

本项目将严格贴合模板结构，主要职责如下。

### 5.1 `src/index.ts`

职责：

1. 初始化全局状态。
2. 注册 WebUI 页面和静态资源。
3. 注册插件内部 API 路由。
4. 启动报告清理定时器。
5. 在卸载时清理定时器和资源。

额外变化：

1. 增加报告目录静态映射。
2. 在初始化时确保 `reports/`、`cache/` 等数据目录存在。

### 5.2 `src/handlers/message-handler.ts`

职责：

1. 扫描群消息中的藏宝阁链接。
2. 校验群级启用状态和全局开关。
3. 执行频控和短时去重。
4. 发送“正在分析”中间态消息。
5. 调用分析编排服务。
6. 发送最终图片或错误提示。

行为约束：

1. 首版每条消息仅处理第一个合法链接。
2. 私聊不作为主支持场景。
3. 群里频控命中时只给出简短提示。

### 5.3 `src/services/`

#### `cbg-fetch-service.ts`

职责：

1. 校验藏宝阁链接格式。
2. 拉取藏宝阁详情接口。
3. 拉取辅助远程配置，如区服和式神数据。
4. 对远程异常、超时和结构变化做统一错误封装。

#### `cbg-analyzer-service.ts`

职责：

1. 承接原 Python 项目的分析规则。
2. 将原本“边分析边拼接文本”的方式改造成结构化报告视图模型。
3. 输出供海报模板消费的稳定数据结构。

说明：

1. 结构化视图模型允许后续测试和迭代中继续补充字段。
2. 模板层不得直接依赖旧式字符串拼接逻辑。

#### `report-render-service.ts`

职责：

1. 用模板生成 HTML 数据海报。
2. 调用 Playwright 截图生成 PNG。
3. 控制视口、字体、背景、布局和输出尺寸。

#### `report-storage-service.ts`

职责：

1. 生成 `report_id`。
2. 管理报告 HTML 和 PNG 的存储路径。
3. 生成公开访问 URL。
4. 清理超过 24 小时的报告文件。

约束：

1. 对外只暴露 URL，不暴露磁盘绝对路径。
2. URL 不包含角色名或本地目录名。

#### `report-orchestrator-service.ts`

职责：

1. 组合抓取、分析、渲染和存储步骤。
2. 对上层提供单一入口。
3. 将原始异常转为受控业务错误。

### 5.4 `src/core/state.ts`

新增职责：

1. 管理最近报告索引。
2. 管理清理定时器。
3. 记录最近错误摘要。
4. 暴露数据文件读写能力给报告索引和运行统计使用。

### 5.5 `src/webui/`

角色定义：

1. 不是主要分析入口。
2. 是插件的运营、配置和排障后台。

## 6. 报告视图模型

分析层将产出结构化视图模型，而不是最终拼接字符串。首版建议结构如下：

```ts
interface BenchPosterViewModel {
  reportId: string
  sourceUrl: string
  generatedAt: string
  hero: {
    areaName: string
    serverName: string
    equipName: string
    priceText: string
    statusText: string
    daysText: string
  }
  highlights: string[]
  resources: Array<{ label: string; value: string }>
  collection: {
    ownedSp: number
    ownedSsr: number
    linkageSummary: string[]
    skinSummary: string[]
  }
  overview: {
    speedSummary: string
    critSummary: string
    inventorySummary: string[]
    suitJudgements: string[]
  }
  pve: {
    headers: ['土蜘蛛', '荒骷髅', '鬼灵歌伎', '平均值']
    rows: Array<{
      soulName: string
      values: Array<{
        heroName: string
        metricText: string
      }>
    }>
  }
  speed: {
    sections: Array<{
      title: string
      rows: Array<{
        label: string
        totalText: string
        slotTexts: string[]
        extraText?: string
      }>
    }>
  }
}
```

说明：

1. 这是当前阶段的结构化边界，不是对外冻结接口。
2. `collection` 中展示的是已收集数量，不再沿用“缺失”语义。
3. `overview` 用于承接总览区的高价值摘要，避免和下方精细表格重复。
4. `pve` 和 `speed` 是本次新增的两个核心模块，模板只消费结构化结果，不直接解析藏宝阁原始字段。

### 6.1 `PVE 数据预览` 模块定义

1. 该模块位于单张长海报的中段，以表格卡片方式展示。
2. 首版固定四列：`土蜘蛛`、`荒骷髅`、`鬼灵歌伎`、`平均值`。
3. 每一行表示一个固定的 PVE 评估场景，对应一个明确的御魂方向。
4. 单元格只展示两项：`式神名` 与 `指标值`，不在海报中额外附带说明文字。
5. 指标值统一采用旧项目已存在的攻暴指标口径，而不伪装成实际伤害值。
6. `平均值` 明确定义为前三列中当前可计算指标的算术平均值，若列缺失则按可用列求平均。

### 6.2 `一速预览` 模块定义

1. 该模块位于单张长海报的下段，以细表形式展示。
2. 首版固定四个区块：
   1. `散件一速`
   2. `散件命中一速`
   3. `散件抵抗一速`
   4. `套装一速`
3. 每个区块按行展示不同组合结果，至少包含：
   1. 组合名
   2. 总一速
   3. 六个位置的拆分值
   4. 命中或抵抗等额外结果字段（仅在对应区块展示）
4. `散件一速` 与指定套装一速的计算优先复用旧项目可确认算法；命中一速与抵抗一速沿用同一套“按位置取最优组合”的严格计算方式扩展。

## 7. 数据海报设计

视觉方向：数据海报。

设计要求：

1. 更像可分享的成品图，而不是后台截图。
2. 顶部需要突出账号身份、价格、状态和几项核心结论。
3. 中部展示 `PVE 数据预览`，下部展示 `一速预览`。
4. 视觉风格优先使用强对比背景、明显层次和可读大字信息。
5. 不沿用原项目的纯文本黑底图方案。
6. 维持单张长海报输出，不拆成多图。
7. 海报不展示“注意事项”说明区，也不保留右下角品牌文案。

技术路线：

1. HTML 模板输出。
2. Playwright 截图输出 PNG。
3. 模板静态资源和字体由插件包内提供。

### 7.1 单张海报分区

1. 顶部为账号总览区，展示基础身份、价格、状态、少量亮点标签和核心资源卡片。
2. 中段为 `PVE 数据预览` 表格区，以固定列的方式展示多个场景结果。
3. 下段为 `一速预览` 表格区，以多分区细表展示散件与套装速度能力。
4. 原总览中的御魂摘要区需要压缩并并入新模块，避免同一指标重复出现。

## 8. 报告存储与访问

### 8.1 存储策略

建议目录位于插件数据目录下：

1. `reports/images/` 存放 PNG。
2. `reports/html/` 存放渲染中间 HTML。
3. `reports/index.json` 存放最近报告索引。

### 8.2 访问策略

1. 对外通过插件静态资源路径暴露图片。
2. API 和 WebUI 返回 URL，不返回磁盘绝对路径。
3. 群聊发送图片时使用该 URL 或上传后的图片消息。

### 8.3 清理策略

1. 默认保留 24 小时。
2. 插件初始化时启动定时清理任务。
3. WebUI 提供手动触发清理能力。
4. 清理任务同步删除图片、HTML 和索引记录。

## 9. 配置设计

配置字段建议如下。

```ts
interface PluginConfig {
  enabled: boolean
  debug: boolean
  autoParseLinks: boolean
  commandPrefix: string
  cooldownSeconds: number
  requestTimeoutMs: number
  maxRenderMs: number
  reportRetentionHours: number
  maxRecentReports: number
  groupConfigs: Record<string, GroupConfig>
}
```

群级配置首版保留：

```ts
interface GroupConfig {
  enabled?: boolean
}
```

说明：

1. `commandPrefix` 可以暂时保留给未来扩展命令，如手动帮助、重试或清理。
2. 主入口仍然是自动扫描链接，而不是命令调用。

## 10. WebUI API 设计

首版内部 API 面向插件自带 WebUI，采用模板已有的无鉴权路由风格。

### 10.1 保留并扩展

1. `GET /status`
2. `GET /config`
3. `POST /config`
4. `GET /groups`
5. `POST /groups/:id/config`

### 10.2 新增

1. `GET /reports`
   返回最近生成的报告列表。
2. `POST /reports/cleanup`
   手动触发一次过期报告清理。

### 10.3 不新增

1. 不新增 WebUI 输入链接直接分析的主业务接口。
2. 不对外提供完整 JSON 报告接口。

## 12. 错误处理

群聊对用户仅返回受控文案，不返回以下内容：

1. 本地路径。
2. 绝对目录。
3. 调用栈。
4. 原始异常对象字符串。

用户可见错误分类：

1. 链接无效。
2. 藏宝阁请求失败。
3. 当前数据暂不兼容。
4. 海报生成失败。

日志保留更细错误码与原始异常，供维护时排查。

## 13. 从旧 Python 项目迁移的边界

保留的核心能力：

1. 藏宝阁抓取思路。
2. 账号分析规则。
3. 报告内容组织方式。

放弃的旧能力：

1. CLI 参数解析。
2. 本地交互输入。
3. Pillow 文本直接转图。
4. EXE 打包逻辑。

迁移原则：

1. 不直接嵌 Python 子进程。
2. 不机械翻译全部旧实现。
3. 优先迁移规则和结果，而不是迁移旧代码结构。
4. 对于旧项目已有明确定义的指标，优先精确迁移，不重新发明口径。
5. 对于旧项目没有现成表格实现的新模块，需要先定义结构化口径，再进入实现。

### 13.1 本次新增模块的迁移口径

1. `一速预览`
   1. `散件一速` 与 `招财/指定套装一速` 以旧项目 `bench_speed` 为主要参考来源。
   2. 单颗御魂速度值的口径沿用旧项目标准化后的主属性、副属性和固有属性聚合逻辑。
   3. `散件命中一速` 与 `散件抵抗一速` 复用同样的组合搜索思想，但分别增加命中或抵抗结果展示。
2. `PVE 数据预览`
   1. 不复用旧项目“这个没问题 / 可能做不了”的自然语言结论。
   2. 改为输出结构化数值表，指标核心仍沿用旧项目的攻暴计算口径。
   3. 行场景选择遵循“当前账号、当前藏宝阁数据可稳定算出”的原则，不加入无法由现有数据稳定支持的条目。

## 15. 风险与后续补充点

1. 藏宝阁远程字段可能持续变化，抓取层必须具备容错和日志能力。
2. 原 Python 项目中的部分常量已经过时，迁移时需要按最新样本逐步校准。
3. 结构化视图模型在后续测试中很可能继续补字段，本设计允许这类增量演进。
4. 如果未来要对外提供 HTTP API，可优先复用 `report-orchestrator-service`，而不是重做分析层。

## 16. 本次结论

本项目将基于 `napcat-plugin-yys-cbg-bench` 模板，重构为一个 NapCat 原生 TypeScript 插件：

1. 以群聊自动识别藏宝阁链接并回图为主业务入口。
2. 以 WebUI 作为插件管理后台。
3. 以结构化视图模型作为分析层和渲染层的边界。
4. 以 HTML 数据海报 + Playwright 截图作为最终报告输出方式。
5. 以 24 小时报告清理和不暴露本地路径为安全约束。
